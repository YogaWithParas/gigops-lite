# GigOps Lite v0.3d Spec: Task Assignment Action

## Purpose

Wire one assignment action from the tasks page to the standalone backend while keeping the frontend safe in mock fallback mode.

## Scope

- Add a small frontend API client function for task assignment.
- Wire only the existing assignment action on the tasks page.
- Update local frontend task state after a successful assignment.
- Keep the existing tasks page layout intact.

## Non-Goals

- No backend contract changes unless strictly required.
- No assignment workflow changes on other pages.
- No database, auth, deployment, queueing, or observability additions.
- No persistence beyond the existing in-memory backend behavior.
- No redesign of the tasks page.

## Runtime Behavior

- The tasks page continues to load tasks through the configurable API client.
- Clicking Assign uses the current top-ranked worker as the assignment target.
- The page shows a minimal loading state during assignment.
- The page shows a minimal error message when assignment fails.
- After success, local task state updates so the UI reflects the assignment immediately.

## Backend-Mode Behavior

- If NEXT_PUBLIC_API_BASE_URL is set, clicking Assign calls POST /tasks/:id/assign on the standalone backend.
- The request sends JSON with the chosen agentId.
- The frontend normalizes the backend response into the existing TaskItem shape.

## Mock-Fallback Behavior

- If NEXT_PUBLIC_API_BASE_URL is absent, the frontend does not call the standalone backend.
- The tasks page remains usable in demo mode.
- Assignment updates only local frontend state in fallback mode.
- This keeps Vercel-safe behavior intact without requiring backend deployment.

## In-Memory Backend Limitation

- Standalone backend assignment state is in memory only.
- Assignment changes reset when the backend process restarts.
- This limitation must remain explicit in testing and review.

## Manual Test Checklist

- With no NEXT_PUBLIC_API_BASE_URL set:
  - /tasks loads successfully.
  - Clicking Assign updates the selected task in local UI state.
  - No request is sent to the standalone backend.
- With NEXT_PUBLIC_API_BASE_URL set to the local backend:
  - /tasks loads successfully.
  - Clicking Assign calls POST /tasks/:id/assign.
  - The selected task reflects the assigned state after success.
- In both modes:
  - No frontend crash occurs.
  - Re-score remains unchanged.
  - No unrelated pages are modified.

## Success Criteria

- The tasks page can perform one assignment action in backend mode.
- The tasks page stays safe and usable in mock fallback mode.
- The UI reflects assignment success without a full page reload.
- Errors are surfaced with a readable message.

## Known Limitations

- Backend task details remain thinner than frontend synthetic task details.
- Fallback mode assignment is still demo behavior and not persisted.
- Backend mode remains non-persistent across backend restarts.
